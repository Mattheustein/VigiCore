# VigiCore Limitations

## 1. Scope Limitations

### Detection Only (IDS)
VigiCore is strictly an **Intrusion Detection System**, not an Intrusion Prevention System (IPS).
*   **What it does**: Alerts on suspicious behavior (logs).
*   **What it does NOT do**: Block IPs, kill processes, or modify firewall rules.
*   **Implication**: Remediation must be performed manually by the administrator.

### Log-Based Blind Spots
VigiCore relies entirely on logs generated by the OS and applications.
*   **Rootkits**: Sophisticated kernel-level rootkits can suppress logging, making them invisible to VigiCore.
*   **Network Packet Inspection**: VigiCore does not inspect raw network packets (PCAP). It cannot detect attacks that do not generate logs (e.g., some lower-level DoS attacks or undetected buffer overflows).

## 2. Technical Constraint

### Single-Node Architecture
*   **Design**: All components (Filebeat, Logstash, Elasticsearch, Kibana) run on a single machine.
*   **Risk**: High resource consumption. Elasticsearch is memory-intensive. A system under heavy attack might become unresponsive, causing log loss.
*   **Scalability**: Not designed for high-availability production environments without splitting components across servers.

### Latency
*   **Real-time**: While "near real-time", there is a delay (seconds) between event generation and visualization due to the Filebeat -> Logstash -> Elasticsearch indexing path.

## 3. Future Improvements

*   **Machine Learning**: Integrate Elastic ML features to detect anomalies without hardcoded rules (requires paid license or OpenSearch).
*   **Alerting**: Configure email or Slack notifications for high-priority alerts (requires Elastic Watcher or Kibana Alerting).
*   **Cloud Migration**: Move the ELK stack to a separate server or cloud instance to protect the logs if the host is compromised.
*   **Endpoint Security**: Replace Filebeat with Elastic Agent/Endpoint Security for richer telemetry beyond just logs.
